1. Why Structured Content? (And What Happens Without It)

The phrase Sanity uses is "structured content." It sounds like jargon, but it's actually the whole reason this app works the way it does — and understanding it is what separates a project that scales from one that falls apart.

What "structured" actually means

Most apps store content as blobs — a big chunk of HTML, a Markdown file, a wall of text. That's fine for a blog. But this app needs to do things with clothing data: filter by season, sort by category, send inventory to an AI, write AI results back to specific fields. You can't do any of that with a blob.

Structured content means every piece of data has a defined shape. Each clothing item isn't just a description — it's a document with discrete, named fields:

{
  "name": "Ivory Silk Blouse",
  "category": "tops",          // not "Top" or "T" — exactly "tops"
  "color": "ivory",
  "season": ["spring", "summer"],
  "occasions": ["casual", "brunch"],
  "image": { "_ref": "image-abc123-800x600-jpg" }
}

Because of this structure, the AI prompt can say "category must be one of: tops, bottoms, dresses..." and trust that Claude's response will be queryable. The frontend can filter items by season without parsing anything. The packing planner can group items by occasion without writing custom logic. Structure is what makes the AI integration clean instead of fragile.

What you'd deal with if you stored this as unstructured data

Imagine instead you just kept notes in a Google Doc, or stored everything as a block of text per item. The moment you want to ask "give me all the tops that work for fall," you're stuck — there's nothing to filter on. You'd send the entire raw text dump to Claude every time and hope it figures out what's a top vs. a dress. That's slow, expensive, and unreliable.

Or say you stored it as a flat JSON file. That's more structured, but now:

  • Adding a new item means editing a file, committing it, and deploying — just to add a shirt
  • When Claude auto-tags an item, there's nowhere to write the result back to. There's no concept of "update this document's field." You'd be building a mini-database from scratch
  • Images have nowhere to live — you'd have to manage S3, resizing, CDN yourself
  • If a friend wants to help catalog items, they're editing raw JSON

Sanity is just a very good place to put structured content

The thing Sanity does is give you a proper home for that structure. You define the shape once in TypeScript — that's the schema. Sanity reads it and:

  • Generates the Studio editing UI automatically (dropdowns, checkboxes, image upload — all from your field definitions)
  • Stores the data in a managed database called the Content Lake
  • Hosts the images on a CDN with on-the-fly resizing
  • Lets you query it with GROQ — one line to filter, sort, and join across document types

The auto-tag feature is the payoff. When Claude returns { category: "tops", season: ["spring"] }, we call patch.execute([{ set: tags }]) and those exact values land in the exact fields Sanity expects. That only works because the schema defined what those fields are, and the prompt was written to match them. The structure connects everything.

One-line summary for the video: Unstructured content is for humans to read. Structured content is for code to use. The moment you want AI to do something useful with your data — filter it, reason about it, write back to it — you need structure. Sanity is just the cleanest way to get there.

2. What We Built

MAISON is a digital wardrobe app for a fashion brand. It solves a real problem: when you own hundreds of clothing items, figuring out what to wear is hard. The app does three things:

  1. Catalog — Upload photos of every piece in your closet into Sanity CMS. Claude Vision auto-fills the name, category, color, seasons, and occasions for you.
  2. Recommend — Describe an event ("rooftop dinner in Miami") and get a styled outfit recommendation pulled from your actual inventory.
  3. Pack — Tell it where you're going and how many days, and it generates a complete packing plan with daily outfits.

The whole point of building on top of Sanity is that your clothing inventory lives in a structured content database — not a flat JSON file, not a spreadsheet. This means you get Studio UI for free, GROQ for querying, image hosting on Sanity's CDN, and the ability to add custom AI-powered workflows directly into the editor.

Step 1 — Sanity Schema

Files: src/sanity/schemas/clothingItem.ts, src/sanity/schemas/curatedOutfit.ts, src/sanity/schemas/index.ts

The schema defines your data model. Sanity reads these TypeScript objects and automatically generates the Studio UI — every field becomes a form input. No React needed for the CMS interface.

clothingItem.ts

import { defineField, defineType } from "sanity";

export const clothingItem = defineType({
  name: "clothingItem",      // used in GROQ: *[_type == "clothingItem"]
  title: "Clothing Item",    // label shown in Studio sidebar
  type: "document",          // top-level content type
  fields: [
    defineField({ name: "name", type: "string", validation: r => r.required() }),
    defineField({
      name: "category",
      type: "string",
      options: {
        list: [
          { title: "Tops",       value: "tops"       },
          { title: "Bottoms",    value: "bottoms"    },
          { title: "Dresses",    value: "dresses"    },
          { title: "Outerwear",  value: "outerwear"  },
          { title: "Shoes",      value: "shoes"      },
          { title: "Accessories",value: "accessories"},
        ],
      },
    }),
    defineField({ name: "color", type: "string" }),
    defineField({
      name: "season",
      type: "array",
      of: [{ type: "string" }],
      options: {
        list: ["spring","summer","fall","winter"].map(s => ({ title: s, value: s })),
      },
    }),
    defineField({
      name: "occasions",
      type: "array",
      of: [{ type: "string" }],
      options: {
        list: ["casual","formal","business","party","outdoor","date-night","wedding","brunch"]
          .map(o => ({ title: o, value: o })),
      },
    }),
    defineField({ name: "description", type: "text", rows: 3 }),
    defineField({ name: "image", type: "image", options: { hotspot: true }, validation: r => r.required() }),
    defineField({ name: "price", type: "number" }),
  ],
  preview: {
    select: { title: "name", subtitle: "category", media: "image" },
  },
});

Critical: The value strings in options.list (e.g. "tops", "casual") must exactly match what you put in your Claude prompt later. If they're out of sync, Claude might return "Top" and Sanity will reject it.

curatedOutfit.ts

import { defineField, defineType } from "sanity";

export const curatedOutfit = defineType({
  name: "curatedOutfit",
  title: "Curated Outfit",
  type: "document",
  fields: [
    defineField({ name: "name", type: "string", validation: r => r.required() }),
    defineField({
      name: "occasion",
      type: "string",
      options: {
        list: ["casual","formal","business","party","outdoor","date-night","wedding","brunch"]
          .map(o => ({ title: o, value: o })),
      },
    }),
    defineField({
      name: "items",
      type: "array",
      of: [{ type: "reference", to: [{ type: "clothingItem" }] }],
      validation: r => r.min(2).max(6),
    }),
    defineField({ name: "curatorName", type: "string" }),
    defineField({ name: "notes", type: "text", rows: 3 }),
  ],
});

schemas/index.ts — Register both schemas

import { clothingItem } from "./clothingItem";
import { curatedOutfit } from "./curatedOutfit";

export const schemaTypes = [clothingItem, curatedOutfit];

Easy mistake: If you create a schema but forget to add it to schemaTypes, Studio will not show it. This is the most common "why doesn't my schema appear" issue.

Step 2 — Sanity Client + Image URLs

Files: src/sanity/client.ts, src/sanity/image.ts

client.ts — how your Next.js app talks to Sanity

import { createClient } from "next-sanity";

export const client = createClient({
  projectId: process.env.NEXT_PUBLIC_SANITY_PROJECT_ID!,
  dataset:   process.env.NEXT_PUBLIC_SANITY_DATASET || "production",
  apiVersion: "2024-01-01",
  useCdn: false,  // false = always fresh data (no CDN cache)
});

image.ts — why this file exists

Sanity does not store images as URLs. It stores them as references like image-abc123def456-800x600-jpg. This is intentional — it lets Sanity serve optimized images at any size on demand from their CDN.

But Claude needs an actual https:// URL to look at an image. So you need to convert the reference into a URL first.

import imageUrlBuilder from "@sanity/image-url";
import { client } from "./client";

const builder = imageUrlBuilder(client);

// Usage: urlFor(item.image).width(400).url()
// Returns: https://cdn.sanity.io/images/projectId/dataset/abc123.jpg?w=400
export function urlFor(source: any) {
  return builder.image(source);
}

The URL builder chains transforms before generating the final URL:

  • .width(400) — resize to 400px wide
  • .height(400) — resize to 400px tall
  • .quality(70) — compress to 70% quality
  • .format("webp") — convert to WebP
  • .url() — finalize and return the string

Step 3 — GROQ Queries

File: src/sanity/lib/queries.ts

GROQ (Graph-Relational Object Queries) is Sanity's query language. Think of it like SQL but for JSON documents. You write a filter expression and a projection (which fields to return).

// Fetch all items, newest first
export const ALL_CLOTHING_QUERY = `
  *[_type == "clothingItem"] | order(_createdAt desc) {
    _id, name, category, color, season, occasions, description, image, price
  }
`;

// Fetch curated outfits AND resolve the clothing item references
// The -> operator is a JOIN — it follows the reference and returns the full document
export const CURATED_OUTFITS_QUERY = `
  *[_type == "curatedOutfit"] {
    _id, name, occasion, curatorName, notes,
    "items": items[]-> {
      _id, name, category, color, season, occasions, description, image
    }
  }
`;

// Filter by category using a $parameter (prevents injection, enables reuse)
export const CLOTHING_BY_CATEGORY_QUERY = `
  *[_type == "clothingItem" && category == $category] | order(_createdAt desc) {
    _id, name, category, color, season, occasions, description, image, price
  }
`;

How these queries are used

// In a Server Component (no useEffect needed — runs on the server):
const items = await client.fetch(ALL_CLOTHING_QUERY);

// With parameters:
const tops = await client.fetch(CLOTHING_BY_CATEGORY_QUERY, { category: "tops" });

// Fetching multiple queries in parallel:
const [items, outfits] = await Promise.all([
  client.fetch(ALL_CLOTHING_QUERY),
  client.fetch(CURATED_OUTFITS_QUERY),
]);

Why three separate queries? Each is optimized for a different use case: ALL_CLOTHING_QUERY feeds the homepage grid and the AI prompt. CURATED_OUTFITS_QUERY needs the reference-resolution (->) because outfits store item IDs, not the full objects. CLOTHING_BY_CATEGORY_QUERY is for when you want to filter without fetching everything.

Step 4 — Embedded Studio + Custom Actions

Files: src/app/studio/[[...tool]]/page.tsx, sanity.config.ts, src/sanity/actions/autoTagAction.ts

How we added the "Auto-Tag with AI" button to Studio

When you use the default Sanity hosted studio (the one at yourproject.sanity.studio), you get whatever Sanity ships out of the box. You can edit content, upload images, publish documents — but you can't add your own buttons or custom behavior to the editor. It's a black box.

Embedding the studio inside your Next.js app changes that. Instead of Sanity hosting the studio on their servers, you host it — it lives at /studio in your own app. And because it's running inside your codebase, it uses your sanity.config.ts, which means you can configure it however you want.

Sanity has a concept called Document Actions — these are buttons that appear in the editor toolbar for any document type. By default you get Publish, Unpublish, Delete. But you can register your own. We wrote one called AutoTagAction that adds an "Auto-Tag with AI" button specifically to clothing item documents.

When clicked, it:

  1. Grabs the uploaded image from the current document
  2. Sends it to our /api/auto-tag route, which calls Claude Vision
  3. Takes the JSON Claude returns and writes it directly back into the document's fields

This is only possible because the studio is embedded. The hosted studio has no idea our Next.js app even exists, let alone its API routes. Embedding is what connects the editor to your backend.

The pattern generalizes. You could use the same approach to add any AI-powered button to any document type — "Generate SEO description," "Translate to Spanish," "Suggest related products." Any time you want the content editor to trigger AI, embedded studio + Document Actions is how you wire it up.

sanity.config.ts — registering the custom action

import { defineConfig } from "sanity";
import { structureTool } from "sanity/structure";
import { visionTool } from "@sanity/vision";
import { assist } from "@sanity/assist";
import { schemaTypes } from "@/sanity/schemas";
import { AutoTagAction } from "@/sanity/actions/autoTagAction";

export default defineConfig({
  projectId: process.env.NEXT_PUBLIC_SANITY_PROJECT_ID!,
  dataset: process.env.NEXT_PUBLIC_SANITY_DATASET || "production",
  basePath: "/studio",
  plugins: [structureTool(), visionTool(), assist()],
  schema: { types: schemaTypes },

  document: {
    // Add our custom button to clothingItem documents only
    actions: (prev, context) => {
      if (context.schemaType === "clothingItem") {
        return [AutoTagAction, ...prev];
      }
      return prev;
    },
  },
});

autoTagAction.ts — the "Auto-Tag with AI" button

import { useDocumentOperation, useClient } from "sanity";
import { useState, useCallback } from "react";
import imageUrlBuilder from "@sanity/image-url";

export function AutoTagAction(props: any) {
  const { draft, published, id, type, onComplete } = props;
  const doc = draft || published;

  // useDocumentOperation gives us the "patch" operation —
  // the ability to write fields back into this document
  const { patch } = useDocumentOperation(id, type);
  const client = useClient({ apiVersion: "2024-01-01" });
  const [isRunning, setIsRunning] = useState(false);

  const onAutoTag = useCallback(async () => {
    if (!doc?.image?.asset?._ref) {
      alert("Please upload an image first.");
      return;
    }
    setIsRunning(true);
    try {
      // 1. Convert the Sanity image ref to a CDN URL
      const imageUrl = imageUrlBuilder(client).image(doc.image).width(1024).url();

      // 2. Send to our API route, which calls Claude Vision
      const res = await fetch("/api/auto-tag", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ imageUrl }),
      });
      const tags = await res.json();

      // 3. Write all AI-generated values back into the document fields
      patch.execute([{ set: {
        name: tags.name,
        category: tags.category,
        color: tags.color,
        season: tags.season,
        occasions: tags.occasions,
        description: tags.description,
      }}]);
    } finally {
      setIsRunning(false);
      if (onComplete) onComplete();
    }
  }, [doc, client, patch, onComplete]);

  return {
    label: isRunning ? "Analyzing..." : "Auto-Tag with AI",
    disabled: isRunning || !doc?.image?.asset?._ref,
    onHandle: onAutoTag,
    tone: "primary" as const,
  };
}

Step 5 — Auto-Tag API Route

File: src/app/api/auto-tag/route.ts

This is the bridge between Studio and Claude. The action sends an image URL; this route forwards it to Claude's Vision API and returns structured JSON.

import { NextRequest, NextResponse } from "next/server";
import Anthropic from "@anthropic-ai/sdk";

const anthropic = new Anthropic(); // reads ANTHROPIC_API_KEY automatically

export async function POST(request: NextRequest) {
  const { imageUrl } = await request.json();

  const message = await anthropic.messages.create({
    model: "claude-sonnet-4-20250514",
    max_tokens: 1024,
    messages: [{
      role: "user",
      content: [
        {
          type: "image",
          source: { type: "url", url: imageUrl },  // Claude fetches this URL
        },
        {
          type: "text",
          text: `You are a fashion product cataloger. Analyze this clothing item image and return ONLY valid JSON:

{
  "name": "A concise product name",
  "category": "MUST be exactly one of: tops, bottoms, dresses, outerwear, shoes, accessories",
  "color": "The primary color",
  "season": ["Array of: spring, summer, fall, winter"],
  "occasions": ["Array of: casual, formal, business, party, outdoor, date-night, wedding, brunch"],
  "description": "A 2-3 sentence product description"
}

Return ONLY the JSON object, no markdown.`,
        },
      ],
    }],
  });

  const textBlock = message.content.find(b => b.type === "text");
  let json = (textBlock as any).text.trim();

  // Sometimes Claude wraps in ```json fences — strip them
  if (json.startsWith("```")) {
    json = json.replace(/^```(?:json)?\n?/, "").replace(/\n?```$/, "");
  }

  return NextResponse.json(JSON.parse(json));
}

Schema-prompt connection: The category values in this prompt (tops, bottoms, dresses...) must exactly match the value fields in clothingItem.ts. Claude has no awareness of your schema — you have to bridge that gap manually in the prompt text.

How the Schema Powers the AI Recommendation

This is the part worth spending time on in the video, because it's the core argument for using Sanity over anything else.

When Claude recommends an outfit, it isn't generating text from scratch and hoping for the best. It's working from a structured inventory — a list of real objects, each with the exact same fields, in a predictable shape. That structure comes directly from the schema you defined in clothingItem.ts.

Here's what Claude actually receives when the recommendation route runs:

[
  {
    "_id": "abc123",
    "name": "White Linen Blazer",
    "category": "outerwear",
    "color": "white",
    "season": ["spring", "summer"],
    "occasions": ["dinner", "travel"],
    "description": "Relaxed fit, slightly oversized, unlined"
  },
  {
    "_id": "def456",
    "name": "Wide-Leg Cream Trousers",
    "category": "bottoms",
    "color": "cream",
    "season": ["spring", "summer"],
    "occasions": ["dinner", "work", "travel"],
    "description": "High waisted, flowy fabric"
  },
  ...300 more items
]

Because every item has the same shape, Claude can reason across the entire closet systematically. It can filter by season, cross-reference occasions, think about color relationships between specific pieces, and return a response that references items by their actual _id. The app then uses those IDs to look up the full item records — including their Sanity image URLs — to display in the UI.

Compare this to what you'd have if you didn't use structured content:

  • If clothes were stored as flat image files with no metadata, Claude would have to guess what each item was from the photo alone — and it couldn't filter by season, occasion, or color without looking at every image every time
  • If you used a spreadsheet or a loose JSON file, you'd have no editor UI, no image hosting, no query language. You'd be writing custom file-parsing logic just to get the data into a shape Claude can use
  • If you built your own database from scratch, you'd need to design the schema, build the admin UI, set up image storage and CDN, write an API layer — all before you could write a single line of AI logic

Sanity gives you all of that. The schema is the contract that makes the AI useful. Every field you define in clothingItem.ts — category, season, occasions — becomes a dimension the AI can reason across. The more structured the data, the smarter the recommendations feel.

The one-liner version: Unstructured content (raw photos, free-text notes) makes AI guessing. Structured content makes AI reasoning. Sanity is what turns a folder of outfit photos into a database an AI can actually think with.

The schema → prompt connection

There's one subtle but important detail: the values in the schema have to match the values in the AI prompt. When we defined category in the schema, we gave it a fixed list of options: tops, bottoms, dresses, outerwear, shoes, accessories. The auto-tag prompt tells Claude to return one of those exact values. The recommendation prompt tells Claude to filter by those exact values.

If the schema says "outerwear" and the prompt says "jacket", nothing breaks — but the AI starts making up category names that don't match the data, and filtering stops working reliably. The schema is the source of truth. The prompt has to respect it.

Step 6 — Outfit Recommendation API

File: src/app/api/recommend/route.ts

This is the most complex route. It does three things in a loop:

  1. Pick an outfit — sends the full inventory as JSON to Claude, asks it to select item IDs
  2. Visual vibe check — downloads the actual images of selected items and sends them to Claude Vision to verify the outfit looks good
  3. Retry if needed — if the vibe check fails, feeds the feedback back into a new pick attempt (up to 3 attempts)
// Simplified version of the core loop:

const [allItems, curatedOutfits] = await Promise.all([
  client.fetch(ALL_CLOTHING_QUERY),
  client.fetch(CURATED_OUTFITS_QUERY),
]);

// Step 1: Pick outfit from structured data
const pickResponse = await anthropic.messages.create({
  model: "claude-sonnet-4-20250514",
  max_tokens: 1024,
  messages: [{
    role: "user",
    content: `You are a personal stylist. Here is the full inventory:
${JSON.stringify(inventory)}

EVENT: "${eventDescription}"

CURATED OUTFITS (these define the client's taste):
${curatedOutfitsText}

Pick the best outfit. Return JSON: { outfitName, selectedItems: ["id1","id2"], reasoning, stylingTips }`
  }],
});

// Step 2: Download images and send to Claude for visual check
const vibeCheck = await anthropic.messages.create({
  model: "claude-sonnet-4-20250514",
  max_tokens: 512,
  messages: [{
    role: "user",
    content: [
      { type: "text", text: "Do these pieces look good together visually?" },
      ...itemImages,   // base64 images of each selected piece
      ...inspoImages,  // reference images of outfits the client loves
    ],
  }],
});

Why the vibe check matters: Claude can pick items that make logical sense ("navy blazer + white shirt = business casual") but look bad together when you actually see the photos. The visual check catches mismatched textures, clashing tones, and proportion problems that text descriptions miss.

Step 7 — Frontend UI

Files: src/app/page.tsx, src/components/ClosetGrid.tsx, src/components/OutfitRecommender.tsx, src/components/OutfitResult.tsx

The homepage is a Next.js Server Component that fetches clothing data at request time and passes it to the grid:

// app/page.tsx — runs on the server, no useEffect needed
export default async function Home() {
  const items = await client.fetch(ALL_CLOTHING_QUERY);
  return (
    <main>
      <ClosetGrid items={items} />      {/* displays the browseable collection */}
      <OutfitRecommender />             {/* client component with the AI chat UI */}
    </main>
  );
}

The OutfitRecommender is a Client Component because it has interactive state (input field, loading state, result display). It calls /api/recommend and renders the OutfitResult when done.

The OutfitResult component:

  • Shows the outfit name and reasoning
  • Displays selected items in a horizontal strip with their photos (from Sanity CDN)
  • Opens a lightbox modal on item click
  • Auto-triggers the /api/illustrate endpoint to generate a DALL-E fashion illustration
  • Shows the illustration as a hero image above the item strip

Bonus Features

Trip Packing Planner

The /api/pack route takes a destination, number of days, and travel style. It fetches the full inventory and asks Claude to plan daily outfits. Important: for trips longer than ~14 days, you need max_tokens: 4096 or higher, otherwise the response gets cut off mid-plan.

Friend Curation Page (/curate)

Password-protected page where stylish friends can browse the inventory and hand-pick outfits as "curatedOutfit" documents. These curated outfits are fed into the recommendation prompt as taste references — the most reliable way to improve AI recommendation quality.

Bulk Upload Script

The script at scripts/bulk-upload.mjs takes a folder of JPEG/PNG images, uploads each to Sanity, creates a draft clothingItem document, calls the auto-tag API, and publishes. Usage:

node scripts/bulk-upload.mjs ~/Desktop/clothes-photos

Note on HEIC images: iPhone HEIC format doesn't work with Sanity's upload API. Convert to JPEG first (or just screenshot the photo as a PNG).

Fashion Illustrations (OpenAI DALL-E 3)

After an outfit recommendation loads, OutfitResult automatically calls /api/illustrate with the outfit description. This hits OpenAI's DALL-E 3 API and returns a fashion illustration URL. Note: ChatGPT Pro subscriptions do not include API credits — you need a separate OpenAI API billing account at platform.openai.com.

Key Talking Points for the Video

  • "Sanity is the database, Studio is the editor, GROQ is the query language — and you get all three for free."
  • "The schema is your contract. It defines what Claude can return." — the category values in the schema and in the prompt must match.
  • "Embedded Studio unlocks custom workflows." The hosted studio can't run your custom document actions. Embedding it in Next.js gives you full control.
  • "Image refs aren't URLs." Sanity stores images as references. You need imageUrlBuilder to get actual URLs for the CDN and for Claude Vision.
  • "Two calls to Claude, not one." The first call picks pieces from structured data. The second call does a visual check by actually looking at the photos. This dramatically improves quality.
